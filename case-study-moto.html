UX Design · Automotive · Motorcycle Interface

Complex Systems. Simple Rides.

I worked as UX Designer on BMW Motorrad's next-generation TFT digital instrument cluster, part of an 8-person UX/UI team developing an adaptive system for urban, sport, and adventure motorcycles launching in 2027.

For more than 1 year, I focused on system architecture and rider-centered features: creating a comprehensive flowchart mapping the entire interface structure, developing decision trees for dynamic features like heating and ride height, designing priority matrices for safety-critical alerts, and leading the customization system that gives riders one-tap access to critical vehicle data.

Context & Constrains

Between Safety,
Complexity and Speed

Designing for motorcycles means designing for life-or-death contexts. Every interaction happens while riders navigate traffic at speed, wearing helmets. Any distraction can be fatal.

The instrument cluster serves three motorcycle types (urban, sport, adventure) with different feature sets, handles unpredictable sensor data, and adapts to multiple languages within a fixed screen size.

The challenge wasn't designing individual screens - it was architecting a system that flexes across models, features, and failure states while never compromising rider safety.

Design Strategy

Foundation
for Consistency

My role centered on building the structural foundation — system architecture, behavior rules, and safety logic - that would ensure consistency across all interface states while keeping riders safe.

Core principles:

  • System structure flowchart: I created and maintained a comprehensive flowchart documenting navigation paths, popup behaviors, overlay interactions, and shortcuts. Designers used it for transitions; developers for implementation.

  • Decision trees for dynamic features: For heating and ride height, I mapped every state: available/unavailable (based on bike model), active/inactive, levels, and errors.

  • Popup priority matrix: I defined which alerts override others when multiple appear. Safety warnings (critical/high) always come first. Rules specify: blocking vs. non-blocking, queuing vs. simultaneous display.

  • Rider-centered customization: Riders choose critical vehicle data (tire pressure, engine temperature, fuel). One button press access - no menus.
System Structure Flowchart

Mapping
the Entire Interface

As features were being defined across the team, we needed a single reference documenting how everything connected. I created and maintained the system structure flowchart - a living document that mapped the complete interface architecture.

  • Navigation structure: Every screen, every path, how riders move through the interface from main menu to deepest levels.

  • Popup and overlay behavior: Where they open, how they dismiss, and what happens when riders press buttons while a popup is visible — does it persist, update, or close?

  • Shortcuts: Direct access paths that let riders bypass standard navigation, with clear rules for when and how they activate.

The flowchart became the team's source of truth. Designers referenced it for transitions, developers for implementation — ensuring the entire team produced consistent interactions.

Popup Priority Matrix

Alert System
Priority

I created a priority matrix defining which popups take precedence and how they behave when appearing simultaneously. These examples illustrate the core logic applied across all interface alerts.

  • Critical messages (e.g. engine failure): Immediate safety threats. Always interrupt everything, including open dialogues. Require manual dismissal.
  • Warning messages (e.g. tire pressure, ABS): Important alerts. Same behavior as critical — interrupt dialogues and push them to stack — but queue behind any active critical messages.
  • Feedback messages (e.g. setting confirmed): Confirmations and changes. Coexist with other popups - layer alongside dialogues and warnings. Exception: suppressed when critical messages are active.

When critical or warning messages appear during an open dialogue, the dialogue pauses and returns to the stack. Feedback messages display alongside existing popups unless critical alerts demand full attention.

Feature . Settings

Sorting logic -
and edge cases

The Settings screen needed to accommodate different combinations of options depending on market, while remaining clean and comprehensible. The main challenge was sorting logic — and the edge cases it created.

Key improvements:

  • In frequency sort, DAB stations come first (by block, then alphabetically), followed by FM/AM
  • When no Public Value stations exist, it shows a clear message + full list
  • Region-specific toggles (Show AM, HD Radio, Radio Info)
  • Consistent handling of all edge cases, preventing empty screens

This structure ensures global consistency while staying locally relevant.

Feature . Metadata Cascade

Logic When Data
Is Missing

Metadata from radio stations is not guaranteed. Stations may send a full set of information, partial information, or nothing at all. The component needed to behave gracefully across all scenarios - we controlled the display logic, not the data source.

I defined a cascade of fallback rules across two independent dimensions - image and text - ensuring the component always had something meaningful to show and avoiding empty states that users wouldn't perceive as elegant.

Testing & validation

Industry
tests limitations

In automotive UX, dedicated usability testing isn't always feasible — this project had no simulator access due to timeline constraints. Validation used industry standards instead: HMI guidelines, ISO safety compliance, and reviews with Functional Owners + engineering.

Next steps (if possible):

  • Driving simulator tests (cognitive load, glanceability, speed)
  • Independent HMI heuristic evaluation (Manual Tuner focus)
Key Learnings & Impact

Main lessons
from this project

This redesign delivered a production-ready radio player balancing legacy features, market variations, and data challenges within automotive constraints.

  • Data vs. Requirements: When usage data conflicts with stakeholders requirements (like legacy tuning), relocate features to high-traffic areas rather than removing them.

  • Edge Case Mastery: Map market variations (USA/Japan fixed lists vs. Germany Public Value) into tables for clarity, then build systematic fallbacks.

  • Process Documentation: I learned that documenting decisions throughout the process is crucial - it forces clearer thinking about edge cases, trade-offs, and validation alternatives.

  • Stakeholder Collaboration: Translate specific concerns (like mode confusion) into universal design principles that build trust across teams.

  • When simulator testing isn't possible, use HMI/ISO standards and stakeholder reviews as reliable alternatives — and always document your ideal next steps (driving tests and HMI heuristics).

Impact: Zero empty states, consistent global behaviors, and driver-focused interactions — delivering reliability where it matters most, the safety-critical automotive UX.
