One Radio. Every Road.

A next-generation in-car radio experience designed for global markets - balancing legal requirements, legacy features, and a shared component system.

Duration

8 months

Industry

Automotive

Role

UX Designer

Overview

Context & Role

Over 8 months, I was the lead UX designer for the in-car radio feature within a next-generation automotive media system. The project was already underway when I joined - the foundational concept was in place, but a significant scope of work remained: validating that every market's requirements were covered, ensuring the design system had all the necessary component variants, and completing three key features from concept to handoff.

I worked closely with Functional Owners, Product Owners, and developers - owning the radio feature end-to-end, from early requirements gathering through wireframes, prototypes, user flows, and final documentation.

The Challenge

One System.
Many Markets.

Radio is not a universal product. Listening habits, technical standards, and legal requirements vary significantly across markets - and our system needed to handle all of them through a single, cohesive component framework.

"The goal was not to build different versions for each market. It was to design components flexible enough to cover every variation, while maintaining consistency and reducing implementation effort."

This required balancing three forces in tension: delivering the best possible user experience, respecting strict implementation constraints, and satisfying market-specific requirements - including a legal one that arrived mid-project.

Underlying every decision was a non-negotiable constraint: drivers interact with the radio while in motion. Minimising distraction and making interactions as fast and intuitive as possible was not a design preference - it was a safety requirement.

Feature Deep Dive · 01

Public Value -
A German
Legal Requirement

Public Value is a German legal requirement: certain radio stations must be identified as publicly funded. What made this particularly interesting was that the indicator is not static - it appears in Germany and disappears when a driver crosses into Austria or Italy. A requirement that travels with the car, not the market.

The requirement arrived mid-project with a tight legal deadline, and with no prior benchmarks - the regulation was new. The challenge was to meet it without disrupting a component that had to work across all markets.

The Problem

How do you display a label that only exists in one market?

Spelling out "Public Value" was too long and added cognitive load - critical in an in-car context. A standalone icon was too abstract. But the information had to be always present for legally qualifying stations.

The Insight

Progressive disclosure across contexts

Not every screen needs to explain everything. In compact contexts - lists, widgets - the abbreviation PV is sufficient. In the full-screen player and settings, the full label "Public Value" appears alongside the icon, providing comprehension where there's space for it.

The Decision

Icon positioning to prevent layout instability

PV is a transient indicator - it appears in Germany and disappears when the driver crosses into Austria or Italy. To prevent jarring layout shifts, the frequency icon (always present) was anchored to the right, and the PV icon was placed to the left. When PV disappears, nothing moves.

Edge Case

Sorting by PV when no PV stations exist

If a user sorts by Public Value in a market or region with no qualifying stations, the list doesn't break - it surfaces a clear message explaining that no PV stations are available, then shows all available stations below.

The same indicator scales in verbosity: abbreviated in compact contexts, fully labelled where space and attention allow.
Feature Deep Dive · 02

Manual Tuner -
Built Around
Real Usage

Manual tuning - seeking across the spectrum or incrementing/decrementing frequency - existed in the previous generation but was rarely used. The data pointed to why: accessing it meant leaving the main screen entirely. For a feature used while driving, that navigation cost was too high.

Usage data also revealed that the full-screen player was by far the most used view - users spent the majority of their time there. That insight shaped the entire approach: rather than designing a separate tuner experience, the goal was to bring manual tuning into the space users already trusted, with minimal disruption to their mental model.

"Manual tuning is exploratory - it serves a fundamentally different user mental model. That distinction, combined with the imperative to minimise in-car distraction, shaped every decision."

Early Exploration

What if all controls were always visible?

Initial explorations placed seeking, increment/decrement, and standard radio controls together in the full-screen player. The space allowed it technically - but the cognitive load was too high. In a driving context, presenting unfamiliar controls alongside familiar ones increases the risk of hesitation and distraction. Simplicity was not optional.

The Solution

A mode, not a screen

Rather than opening a separate manual tuner screen (as in the previous generation), I proposed a toggle button in the full-screen player. Activating it replaces standard playback controls with manual tuning controls - without navigating away. The user stays within the full-screen layout they already know and trust, retaining access to all station metadata. Fewer navigation steps means less time eyes-off-road.

Stakeholder Concerns

Making the mode change unmistakably clear

Stakeholders were concerned that users wouldn't know they'd entered a different mode. The solution was layered: the toggle icon is visibly selected, the screen title updates, the AM/FM toggle appears exclusively in this mode, and a subtle animation transitions between the two sets of controls - making the shift unambiguous without being disruptive.

Deliverables

Wireframes, prototypes & user flows

I produced wireframes to communicate the Manual Tuner concept clearly to stakeholders and developers. To bring the mode transition to life, I built animated prototypes showing how the controls would switch - making abstract animation descriptions concrete and reviewable. User flows were also documented to map every possible interaction path, giving both stakeholders and developers a clear reference for implementation.

Benchmarking

Informed by industry patterns

I researched how other automotive brands implement seeking and increment/decrement controls. Common patterns existed across the industry - aligning with them reduced the learning curve for drivers already familiar with other systems.

Feature Deep Dive · 03

Settings -
Complexity, Structured

The Settings screen needed to accommodate different combinations of options depending on market, while remaining clean and comprehensible. The main challenge was sorting logic - and the edge cases it created.

01

Market-Specific Sorting Options

Rest of World supports alphabetical and frequency sorting. The German market adds Public Value as a sorting criterion. Each combination required defined behaviour.

02

DAB Stations in Frequency Sort

DAB stations don't map neatly onto FM/AM frequency ordering. Solution: DAB stations appear first, ordered by DAB block, then FM, then AM. Stations sharing the same DAB block are sorted alphabetically.

03

Market-Specific Toggles

Rest of World shows Radio Info and Show AM toggles. US/Japan shows Show AM Stations and HD Radio toggles. Conditional rendering keeps settings relevant to the user's context.

04

PV Sort with No PV Stations

Consistent with the list behaviour: if PV is selected as a sort criterion and no stations qualify, a clear message is shown and all available stations follow below.

Feature Deep Dive · 04

Metadata Cascade -
Logic When Data
Is Missing

Metadata from radio stations is not guaranteed. Stations may send a full set of information, partial information, or nothing at all. The component needed to behave gracefully across all scenarios - we controlled the display logic, not the data source.

I defined a cascade of fallback rules across two independent dimensions - image and text - ensuring the component always had something meaningful to show.

-

Image Hierarchy

Album cover art (best case) → Station logo → Generic fallback image. Each level is independent of the text metadata state.

-

Text Hierarchy

Song title + artist name (best case) → Radio programme name → Station slogan or descriptor. If no station name is available, the frequency is shown instead.

Diagram 4 - Metadata Cascade Logic
Two parallel fallback chains ensure the component always displays something meaningful, regardless of what data the station provides.

All cascade rules were documented and shared with the development team, providing a precise reference for every possible data state.
